# Ship as a container: Caddy serves the static build and same-origin-proxies /api

StringPro is a pure client-side SPA ‚Äî `vite build` emits static files in `dist/` and nothing else. To run it on a VServer (behind a host reverse proxy that terminates TLS) we package it as a container. The build output has no server of its own and, crucially, the `/api` proxy in `vite.config.ts` is a **dev-server-only** feature that does not survive `vite build`. Production therefore needs a real web server inside the image.

Decision: a multi-stage `Dockerfile`. A `node:22-alpine` builder runs `pnpm build` (pnpm pinned via the `packageManager` field + Corepack; `pnpm-workspace.yaml` is copied into the install layer so `allowBuilds` lets sharp's native build run); the resulting `dist/` is copied into a `caddy:2-alpine` runtime stage. Node and `node_modules` are left behind ‚Äî the final image is ~85 MB.

Decision: **Caddy** serves the static files and reverse-proxies same-origin `/api/*` to the public backend `https://stringpro-backend.lausi95.net`, mirroring the dev Vite proxy (`handle_path` strips the `/api` prefix; the upstream `Host` header is set to the backend domain so the host TLS proxy routes it by SNI + vhost). The browser only ever talks to its own origin, so there is **no CORS** and no absolute backend URL baked into the bundle ‚Äî the backend origin is a runtime env var (`BACKEND_ORIGIN`), changeable without a rebuild. This keeps the code's existing `API_BASE = VITE_API_BASE ?? '/api'` default and the PWA `navigateFallbackDenylist: [/^\/api/]` (ADR 0011) valid.

Reason for Caddy over nginx: JSON structured access logs to stdout are the default (one line per request ‚Üí collected by the host Datadog agent, no in-image agent), and reverse-proxy + SPA history fallback + non-root + HTTPS-upstream SNI are each ~1 line. `auto_https off` because TLS is the host proxy's job; Caddy runs rootless on `:8080`.

Decision: **tiered cache headers**, because the autoUpdate PWA (ADR 0011) only picks up a new deploy silently if the entry points revalidate. Content-hashed files (`/assets/*`, the hash-named `workbox-*.js`) get `Cache-Control: public, max-age=31536000, immutable`; stable-named entry points (`index.html`, `sw.js`, `manifest.webmanifest`) get `no-cache`. Getting this wrong pins users to a stale build ‚Äî the exact failure silent auto-update exists to avoid.

Consequence: the image is self-contained and portable (`docker run` anywhere; routing travels with the artifact). The frontend's *browser* console/errors are **not** captured ‚Äî this container's only logs are Caddy's access/error logs; collecting client-side errors would need Datadog RUM (browser SDK), a separate future decision. Verified against a running container (browser, port 5173 so Keycloak accepts the `redirect_uri`): non-root, correct tiered cache headers, SPA deep-link fallback, JSON access logs on stdout, the app boots and renders, the service worker registers, and an **authenticated** `/api/customers` call returns `200` through the proxy with a real Keycloak Bearer token. The GitHub Actions build is deferred; the Dockerfile takes no build args, so wiring CI later is additive.

Known wrinkle (not fixed): the backend's `401` challenge advertised its `resource_metadata` URL over `http`, because the container's Caddy‚Üíbackend hop is plain HTTP and the backend derives the scheme from `X-Forwarded-Proto`. It does not affect the browser flow (keycloak-js already holds the token and never consults RFC 9728 resource metadata), so it is left as-is. If the backend ever emits proto-sensitive redirects in production, forward `X-Forwarded-Proto: https` on the `reverse_proxy` in the Caddyfile.
