# Finishing a Job records Reel exhaustion, and a mid-racket run-out splits a Mono Job

When a Job advances **In Progress → Done**, the Stringer is prompted about the Reel(s) it drew from: which are now empty (→ `USED_UP`), and — for a **Mono** Job whose Reel ran out *after the mains* — which substitute Reel finished the crosses. This is the **second** place Job logic mutates Reel state (the first is Job-start, ADR 0008) and revises two earlier claims: ADR 0008's "Job-start is the *only* New → In Use trigger" and the CONTEXT note that retiring to `USED_UP` is "always a manual act." Both are now qualified — see *Reel exhaustion* and *Reel substitution* in `CONTEXT.md`.

## Context

A Reel is shared across many Jobs and the backend tracks its `state` explicitly (ADR 0004); consumption is a derived view that never mutates state (ADR 0007). Until now the sole Job-driven state change was committing a New Reel to In Use when a Job starts (ADR 0008). But two things only become known when the stringing is actually finished: whether a Reel ran *dry*, and — the hard case — whether a Reel ran dry **partway through a racket**, forcing the rest onto another Reel. Both are recorded at the Done transition because that is when the Stringer knows them.

## Decision

**Prompt at the Done advance, gated like the start-commit dialog (ADR 0008).** The prompt appears only when the Job has at least one `REEL` side (`OWN`-only Jobs advance straight through). Cancel aborts the advance — the Job stays In Progress, nothing changes. There is no path to Done that skips the prompt, but "everything lasted, nothing used up" is a valid no-op that simply advances the stage.

**Exhaustion (both Mono and Hybrid).** For each distinct Reel the Job draws from, the Stringer says whether it is now empty; every Reel marked empty transitions to `USED_UP`. The app never infers depletion from meters consumed — consumption stays derived (ADR 0007); this is a deliberate, prompted assertion (ADR 0004). A Reel already `USED_UP` is skipped (no redundant call). We take the Stringer's word: no validation against other active Jobs that share the Reel (single-Stringer installation; they know their bench).

**Mono substitution (the split).** A Mono Job is asked "did this Reel last the whole racket?"
- **Yes** → then "is it now used up?" → `USED_UP` or no-op.
- **No** (ran out after the mains) → the Stringer picks a **substitute Reel** (strictly another Reel, never `OWN`, chosen from `NEW`/`IN_USE` Reels) to finish the crosses, and the Job is **rewritten from Mono to Hybrid**:
  - `PUT /jobs/{id}` with `hybrid: true`, `mains` = the original (exhausted) Reel, `crosses` = the substitute Reel.
  - **The Customer's agreed price is preserved.** The original `totalStringFee` **F** is split **F/2 on each side**, taken from the Job's existing `totalStringFee` and **not** derived from either Reel's own `stringFee`. This holds the total fixed (the Customer already agreed it) and reuses the existing Hybrid meters math (ADR 0007) — each side draws half its Reel's `metersPerJob`. Per-Reel *Earned* therefore splits half/half, which is also the physically accurate attribution (each Reel strung half the racket). **Implementation imperative:** set both sides' `stringFee` to `F/2` from the original total; do **not** reuse the create form's `reelFeeForSide(reel.stringFee, hybrid)` / `onPickReel` auto-fill for the substitute side — that would make the crosses fee `R2.stringFee/2` and silently break preservation precisely when `R1.stringFee ≠ R2.stringFee`.
  - **Load-bearing backend behavior (confirmed).** All of the above depends on the backend *honouring* the per-side `stringFee` in the `PUT` body rather than recomputing each side's fee from its Reel — **confirmed by the Stringer**: the backend preserves the client-sent per-side fee. (Note ADR 0006 was *not* evidence for this on its own: its empirical test was a Mono, single-Reel, full-fee Job, where "honour client fee" and "recompute from `reel.stringFee`" yield the same number, so it never discriminated the two.) Because the backend honours the sent value, a substitute Reel with a *different* String Fee does not disturb the preserved total, so "allow a different substitute string" and "preserve the agreed fee" coexist. **Verified empirically** during implementation against the live backend: a Mono Job (String Fee €7.50) substituted onto a €10.00/job Reel kept its total at **€17.50** — the sent `F/2 + F/2` was honoured, not recomputed to €8.75; the derived per-Reel *Earned* split €3.75 / €3.75; and `PUT /jobs/{id}` was accepted while the Job was at `IN_PROGRESS`.
  - The exhausted mains Reel → `USED_UP` (definitionally empty; not separately asked). The substitute Reel, if `NEW`, is committed to `IN_USE` (the second New → In Use trigger; same semantics as ADR 0008).
  - An **auto-note** is appended recording that a substitution occurred — the only honest trace, since in persisted data a substituted Job is **indistinguishable from a deliberately-chosen Hybrid Job**. "Reel substitution" names the *act*, not a new Job type.

Substitution is **Mono-only**. A Hybrid Job is never offered a substitute Reel — only the per-Reel used-up questions. A Reel is assumed to run out only at a **side boundary** (after the mains), because the data model has exactly two sides and no finer granularity.

**Atomicity — full compensation, ADR 0008 style.** The calls run job-rewrite/Reel-state first and the **stage advance last**, so a failure is caught before the Job moves. Order for a substitution Done: `PUT` rewrite → commit substitute Reel (`NEW → IN_USE`) → exhausted Reel(s) `→ USED_UP` → `PATCH` stage `→ DONE`. On a failure at any step, compensate in reverse — roll back Reel states to what they were and `PUT` the Job back to its original Mono payload — so the end state is "nothing changed, retry," surfaced as an error toast. (A pure-exhaustion Done is just the `USED_UP` PATCHes then the stage advance, same compensation.)

## Considered alternatives

- **Physical-truth / best-effort** — keep `USED_UP` changes even if a later step fails, on the grounds that an empty Reel is empty regardless of the stage. Rejected for consistency: it splits the action into "reverts" and "doesn't," a bespoke partial-state model that is harder to reason about than ADR 0008's uniform "all or nothing."
- **All fee on the original Reel** (F on mains, €0 on the substitute) — rejected; it would make the substitute Reel show real meters but zero earnings, needs a special case, and misrepresents net-return per Reel. Half/half reuses the Hybrid math unchanged.
- **A distinct persisted "split" Job type / marker** — rejected; the backend has no field for it, and the "allow a different substitute string" rule means it cannot even be inferred from same-vs-different String product. The auto-note carries the history instead.
- **Validating shared-Reel exhaustion** against other active Jobs — rejected as gold-plating that tells a single Stringer what they already know.

## Consequences

This is now the second Job-driven Reel-state mutation and the second `New → In Use` trigger; `CONTEXT.md`'s *Reel commitment*, *Reel exhaustion*, and *Reel substitution* entries are the canonical descriptions, and ADR 0008's "only trigger" wording is superseded by them. A Mono Job can change to Hybrid after creation, so any code assuming `hybrid` is fixed at create time (e.g. edit-mode fee handling) must tolerate it. Revisit if the backend ever performs the exhaustion/substitution server-side as part of the stage transition — at which point this client-side saga and its compensation should collapse into a single call, as noted in ADR 0008.
