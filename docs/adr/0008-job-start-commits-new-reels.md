# Starting a Job commits its New Reels to In Use (atomically)

When a Job enters **In Progress** (the only forward transition from Picked Up), any `NEW` Reel it strings from must move to **In Use**. ADR 0004 and ADR 0007 deliberately kept a Reel's `state` a manual act that Jobs never touch: consumption is a derived, read-only view, and reels are never silently activated or retired from usage. This feature intentionally introduces the *one* exception — Job-start drives a New → In Use Reel transition — and the question was how to do it without violating the spirit of "deliberate manual act."

Decision: gate the In Progress transition on **committing** the Job's New Reels, as a single all-or-nothing act that the Stringer confirms.

- **Trigger / scope.** Only the Picked Up → In Progress transition (in `JobDetailPage.handleAdvance`, the sole path into `IN_PROGRESS`). Gather the distinct `REEL`-type sides (mains + crosses, deduped); `OWN` sides have no Reel. If none of the referenced Reels are `NEW`, advance directly with no prompt — today's behavior. Reels already `IN_USE` need nothing; `USED_UP`/already-`IN_USE` handling beyond "leave untouched" is **deferred** (open question).
- **Confirmation.** If any referenced Reel is `NEW`, one combined dialog lists them all and offers *Not now* / *Mark In Use & start*. Declining changes nothing — the Job stays at Picked Up and the Reel stays New. The Stringer can still start the Job later by marking the Reel In Use manually (Strings page) and re-advancing, or by confirming.
- **Hard gate.** The Job advances **only if** its New Reels end up In Use. A deliberate decline does not advance; this is the intended way to refuse — there is no path that starts a Job while leaving its New Reel New.
- **Ordering + atomicity.** Flip the Reel(s) first (`PATCH /reels/{id}/state` → `IN_USE`), then advance the stage (`PATCH /jobs/{id}/stage`). Any failure — a Reel flip *or* the stage advance — triggers compensation: revert every Reel already flipped back to `NEW`. End state is "nothing changed; try again" (error toast). There is no backend transaction across the two aggregates, so the client owns this compensation. (If a revert PATCH itself fails — a degenerate case — surface it honestly rather than claiming success.)

Reason: the confirmation is what reconciles this with ADR 0004/0007. The Reel state change is still a **deliberate** act — prompted, explicit, refusable — not the silent auto-activation those ADRs ruled out. Reels-first ordering means a failed commitment is caught *before* the Job moves, so the Stringer never lands in a "Job In Progress but its Reel still New" state. The atomic revert keeps the two aggregates consistent despite the lack of a server-side transaction; the alternative (best-effort, leave partial state) was rejected in favor of "either it all happened or nothing did."

Consequence: this is the **only** New → In Use trigger, and it is the only place Job logic mutates Reel state — `CONTEXT.md`'s *Reel commitment* entry is the canonical description. *Reel consumption* (ADR 0007) remains derived and never mutates state, and retiring to `USED_UP` remains a separate manual act. Revisit if the backend ever performs the commitment server-side (atomically, as part of the stage transition), at which point the client gate, the reels-first ordering, and the compensation logic should all be removed in favor of a single call. The deferred `USED_UP`/`IN_USE` behavior should be settled before or alongside that.
