# Modal-based create and edit for all entities

All entity create and edit interactions happen in modals layered over the relevant list or detail page — there are no dedicated `/new` or `/edit` routes. We chose this for consistency across entities (Customer, Racket, and future Job/String/Settings) and because editing is always contextual: the Stringer never wants to lose their place on the underlying page for a handful of fields. Each entity gets its own form-modal component (e.g. `CustomerFormModal`, `RacketFormModal`) sharing a common `Modal` shell, rather than a single generic schema-driven modal, since per-entity validation differences (number ranges, email/phone) make a generic abstraction leaky. This supersedes the original route-based `CustomerEditPage` (`/customers/:id/edit`), which was removed.
